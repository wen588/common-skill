# code-explainer Skill 改进报告

## 一、原技能分析

### 原版结构
原版 SKILL.md 共约 40 行，包含：
- 角色定义（1 句）
- 核心指令（1 段）
- 触发条件（3 条）
- 输出规则（4 条）
- 使用场景（3 条）

### 原版优点
- 核心方向正确：比喻先行、避免术语
- 输出规则清晰（先目标→拆步骤→避免术语→收尾）
- 触发条件覆盖了主要场景

### 原版不足

| 问题 | 影响 |
|------|------|
| **缺乏代码类型区分** | 算法、UI组件、数据管线的解释策略应该不同，但原版"一刀切" |
| **无受众感知** | 默认"完全不懂编程"，无法根据情况调整深度 |
| **输出结构松散** | 只有 4 条原则性规则，无具体模板，导致每次输出格式不一致 |
| **无反模式清单** | 没有指出"不要做什么"，如：不要混合比喻、不要逐行解释 |
| **无质量自查机制** | 无法在输出前自我校验质量 |
| **无交互能力** | 解释完就结束，不会追问用户是否需要进一步说明 |
| **触发描述过于保守** | 只在用户明确要求时才触发，错过很多潜在适用场景 |
| **无示例支撑** | 模型只能靠理解原则来产出，缺少高质量参考样例 |

## 二、改进内容

改进后 SKILL.md 扩展至约 220 行，新增以下模块：

### 1. 代码类型策略（6种）

针对不同代码结构提供不同的比喻领域：

| 代码类型 | 推荐比喻域 | 示例 |
|----------|-----------|------|
| 算法/函数 | 菜谱、工厂 | 递归→套娃，循环→传送带 |
| 数据结构 | 物理存储 | 数组→储物柜，链表→寻宝 |
| 架构模式 | 建筑、组织 | MVC→餐厅，事件驱动→门铃 |
| UI 组件 | 剧场、店铺 | State→心情，Props→规格说明 |
| 数据管线 | 流水线 | 过滤→质检，聚合→打包 |
| 状态机 | 自动售货机 | 状态→模式，转移→触发 |

### 2. 分层输出模板（5层）

强制统一输出结构：
- **Layer 1**: 一句话比喻（必须）
- **Layer 2**: 全景图（2-4 句）
- **Layer 3**: 逐步拆解（每步含"现实对照→代码对照→映射关系"）
- **Layer 4**: 术语桥接（可选）
- **Layer 5**: 一句话总结（必须）

### 3. 反模式清单（8条）

明确禁止的行为，防止常见 AI 解释陷阱：

| 反模式 | 问题 |
|--------|------|
| 先用术语开头 | 立即流失用户 |
| 混合比喻 | 造成困惑 |
| 逐行解释 | 信息过载 |
| 用编程概念解释编程概念 | 用未知解释未知 |
| 解释长度超过代码本身 | 啰嗦 |
| 假设 CS 背景 | 很多用户没有 |
| 解释语法而非逻辑 | 语法可搜索，意图不可 |
| 一致性缺失 | 不同代码类型用不同结构 |

### 4. 质量自查清单（6项）

输出前自查：
- 首句是否避免术语？
- 比喻是否一致？
- 非程序员是否能理解？
- 解释是否短于代码？
- 术语是否在比喻之后引入？
- 是否只覆盖关键步骤？

### 5. 交互能力

- 解释后主动询问是否需要深入
- 处理追问时优先复用已有比喻
- 必要时切换比喻域

### 6. 触发条件增强

新增触发场景：
- 用户粘贴代码但未解释时主动询问
- 用户阅读陌生库/框架代码时
- 用户说"我不懂"、"看不懂"、"这是什么"时
- Code Review 中 reviewer 缺少上下文时

### 7. 高质量示例（3个）

提供完整的高质量输出样例（二分查找、React 组件、数据管线），帮助模型理解输出标准。

## 三、对比评估结果

### 评估设置

使用 3 个测试用例，分别覆盖 3 种不同类型的代码：

| 测试 | 代码类型 | 复杂度 | 测试维度 |
|------|---------|--------|---------|
| binary-search | 算法 | 中等 | 比喻质量、无术语开头、结构完整性 |
| react-effect | UI 框架 | 中等 | 比喻质量、cleanup 解释、listener 模式 |
| sql-join | 查询语言 | 简单 | JOIN 解释、分步说明、无术语开头 |

### 量化结果

```
基础断言通过率（anti-pattern / structure compliance）：
  新版技能: 14/14 = 100%
  旧版技能: 14/14 = 100%
```

### 定性差异分析

尽管基础断言都通过，实际输出质量存在显著差异：

#### 差异 1: 输出结构一致性

**新版输出**（binary-search）：
```
🧠 One-liner: Like looking up a word in a paper dictionary...
🏠 The Big Picture: Imagine you have a phone book...
👣 Walkthrough:
   1. ... → In real life: ... → In code: ...
   2. ...
📝 Summary: ...
```

**旧版输出**（binary-search）：
```
这段代码好比你在翻一本厚厚的电话本...
拆开看代码的每一步：
1. left/right → ...
一句话总结：...
```

> 新版统一了 5 层结构，跨不同代码类型产出格式一致；旧版每次格式各异。

#### 差异 2: 步骤映射清晰度

**新版**每步都有 "In real life → In code" 的显式对照，用户能清晰建立映射。

**旧版**虽然也有对照，但结构不统一，有时步骤说明不够清晰。

#### 差异 3: 防止常见错误

**新版**的反模式清单直接防止了"先甩术语"、"混合比喻"等 8 种常见 AI 解释错误。

例如，测试中旧版在 binary-search 结尾直接使用了术语"二分查找"而未加桥接说明：
```
一句话总结：...这就是"二分查找"。
```
而新版保留在比喻域内结尾：
```
Summary: This is "the dictionary lookup method"...
```

#### 差异 4: 跨代码类型适配

| 测试 | 新版策略 | 旧版策略 |
|------|---------|---------|
| binary-search（算法） | 查字典比喻（算法策略） | 电话本比喻（通用） |
| react-effect（UI） | 门铃传感器比喻（UI 策略） | 门铃比喻（通用） |
| sql-join（查询） | 笔记本+收据比喻（管线策略） | 大客户记录（通用） |

新版针对每种代码类型选择了更精准的比喻域，而旧版使用通用比喻策略。

## 四、改进总结

### 做了哪些改动

| 改动项 | 原版 | 改进版 |
|--------|------|--------|
| 内容量 | ~40 行 | ~220 行 |
| 代码类型策略 | 无 | 6 种差异化策略 |
| 输出模板 | 4 条原则 | 5 层强制结构 |
| 反模式清单 | 无 | 8 条 |
| 质量自查 | 无 | 6 项 checklist |
| 交互能力 | 无 | 追问/深入机制 |
| 触发条件 | 3 条 | 扩展到 8+ 场景 |
| 示例 | 无 | 3 个完整样例 |
| Description | 35 字 | 120 字（含触发指引） |

### 核心提升

1. **从"原则驱动"到"结构+策略+约束"三维驱动**
2. **输出从"每次随缘"到"格式统一、质量可控"**
3. **从"被动响应"到"主动感知触发场景"**
4. **从"解释完即止"到"可追问、可深化"**
5. **从"一刀切"到"按代码类型精准适配比喻域"**
